核心原则

原则 说明
客户端 渲染、输入、发 Command；不做权威玩法判定
服务器 校验、改 SyncVar / Transform，再同步给所有客户端
权威 谁改服务器上的“真相”，所有人看到一致结果
单例 Manager 每场景仅一个激活的 NetworkManager 
  flowchart LR
    C[Client] -->|Command| S[Server]
    S -->|SyncVar / ClientRpc / NetworkTransform| C
    S --> C2[其他 Client]

客户端 / 服务器 / Host

Mirror 没有固定的“客户端 exe / 服务器 exe”两套工程，靠打包方式 + 启动 API 区分角色。

客户端打包

  • 可独立运行的游戏本体
  • 负责：输入、渲染、音效
  • 向服务器同步本地状态（位置、操作），接收服务器下发的全局状态
  • 不参与权威判定（血量、胜负等由服务器算）

服务器形态

1. 无头专用服务器（Headless Server）

项 说明
形态 无图形界面的独立可执行程序
内容 仅网络逻辑 + 权威玩法；不加载渲染/UI
职责 管理连接、鉴权、碰撞/得分/血量、全局状态同步
打包 Build Settings 勾选平台 + Mirror Headless Server

2. Host（客户端兼服务器）

项 说明
形态 复用同一客户端程序，运行时 StartHost()
Host 端 既渲染操作，又在本地跑服务端逻辑
其他客户端 连 Host 的 IP:Port
本地通信 Host 的 Client 与内置 Server 走进程内通道，延迟更低
适用 原型、局域网、小规模房间

启动 API

NetworkManager.singleton.StartServer();  // 专服
NetworkManager.singleton.StartHost();    // 房主
NetworkManager.singleton.StartClient();  // 加入（地址端口在 Transport / Manager 配置）

常用判断

属性 含义
isServer 服务器逻辑为真（Host 也为 true）
isClient 客户端为真（Host 也为 true）
isServer && isClient Host
isOwned 本机拥有该 NetworkIdentity（读输入、发 Command）

玩法逻辑用 isServer 判断，不要只用 isClient，否则 Host 会漏执行。

连接建立流程

传输层（Mirror Transport）

协议 用途
TCP 可靠连接、鉴权、关键状态消息
UDP 位置等实时数据（上层可加重传）

开发者一般不直接写 Socket，通过 Mirror Transport 配置即可。

步骤概览 

  sequenceDiagram
    participant S as Server
    participant C as Client
    S->>S: 1. 绑定 0.0.0.0:端口，StartServer/Host 监听
    C->>S: 2. StartClient，指定 IP:Port
    Note over C,S: 3. TCP 三次握手（Transport 自动）
    C->>S: 4. 应用层握手（版本、可选鉴权）
    S->>C: 5. 连接成功，同步房间/玩家列表
    S->>C: 6. Spawn Player 等初始状态
  1. 服务器启动：绑定 IP（常用 0.0.0.0）+ 端口（如 7777），防火墙放行。
  2. 客户端连接：知晓服务器地址（局域网 IPv4 / 公网 IP 或域名 + 端口映射）。
  3. TCP 三次握手：SYN → SYN+ACK → ACK。
  4. Mirror 应用层握手：协议版本、可选密码/防作弊；失败则断开。
  5. 回调：OnServerConnect / OnClientConnected 等做业务逻辑。
场景 连谁
专服 服务器公网/内网 IP:Port
Host Host 机器局域网 IP:Port
本机测试 localhost

NetworkManager

所有连接、生成、场景切换的入口。

必须遵守

  1. 每场景只有一个激活的 NetworkManager（单例）。
  2. 不要挂在带 NetworkIdentity 的 GameObject 上（加载场景时可能被禁用）。
  3. 一棵物体树只能有一个 NetworkIdentity，挂在父节点；父链上不能再有 Identity。

常用配置

项 说明
Transport Telepathy / KCP 等
Network Address / Port 默认连接地址与端口
Player Prefab 带 NetworkIdentity 的玩家预制体
Offline Scene 未连接时场景
Online Scene 连接后自动加载的场景
Registered Spawnable Prefabs 可网络生成的预制体列表

常用回调（自定义继承 NetworkManager）

  • OnServerConnect / OnServerDisconnect
  • OnClientConnect / OnClientDisconnect
  • OnServerAddPlayer / OnServerReady

SyncVar 与 UI 同步

SyncVar 规则

  • 标记需要同步的字段，仅服务器可改值
  • 客户端自动收到更新
  • [SyncVar(hook = nameof(OnXxxChanged))]：值变化时调用 hook

重要：hook 在客户端收到同步后执行（用于更新 UI），不是在服务器上执行。

名字 / 颜色同步流程 

  sequenceDiagram
    participant O as Owner Client
    participant S as Server
    participant A as All Clients
    O->>S: CmdSetName / CmdSetColor
    S->>S: 校验，写入 SyncVar
    S-->>A: 自动同步
    A->>A: hook 更新 TMP/UI

}

为什么本地直接改 UI 别人看不到？

UI 只是本机显示。必须走：Command → 服务器改 SyncVar → 同步 → hook 改 UI。

常见坑

现象 原因 / 处理
Command 不执行 未 Spawn、从服务器调了 Command、无 Identity
SyncVar 不同步 在客户端直接改字段
hook 没触发 UI hook 写错方法名；或以为在服务器执行
Manager 失效 Manager 挂在 NetworkIdentity 上
别人看不到物体 未 NetworkServer.Spawn、预制体未注册
位置回弹 客户端和服务器同时改 Transform
VR 头身分离 同步了 XR Rig 而非 Player 根
