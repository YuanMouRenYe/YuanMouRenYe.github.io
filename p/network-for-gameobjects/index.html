Netcode for GameObjects 笔记

核心原则

原则 说明
客户端 → 服务器 客户端发 RPC 或请求；不要直接在客户端改“全局权威”状态
服务器 → 客户端 服务器（或权威端）改状态后，用 NetworkVariable / NetworkTransform / ClientRpc 下发
单一 NetworkManager 全场景仅一个；Inspector 调试可配合 Multiplayer Tools（Package Manager）
预制体必须登记 动态 Spawn 的预制体要在 NetworkManager 的 Network Prefabs 列表中

一句话记忆

谁权威，谁改 Transform / NetworkVariable；其他人只读或做表现层插值。

RPC 对照（Mirror / 旧 NGO → 新 API）

旧关键字 新版 SendTo 方向 命名习惯（可选）
[Command] SendTo.Server 客户端 → 服务器 CmdXxx
[ClientRpc] SendTo.Clients 服务器 → 所有客户端（不含纯 Server 进程时视版本而定） RpcXxx
[TargetRpc] SendTo 指定客户端 服务器 → 单个客户端 TargetRpcXxx
— SendTo.ClientsAndHost 服务器 → 客户端 + Host 上的本地玩家 广播常用
— SendTo.Server + RpcParams 可指定目标 ClientId 私聊、踢人等

启动模式与常用 API

模式 API 谁跑逻辑 典型用途
Dedicated Server StartServer() 仅服务器有完整模拟；无本地玩家画面（或无头） 正式服、多房间
Host StartHost() 同一进程既是 Server 又是 Client 开发调试、房主开房
Client StartClient() 只接收同步 + 发 RPC 加入房间的玩家

NetworkBehaviour 里判断“我是谁”

属性 含义 典型用法
IsServer 当前实例在服务器逻辑上为真（Host 也为 true） 扣血、刷怪、判定胜负
IsClient 当前是客户端（Host 也为 true） 播本地特效、UI
IsHost 既是 Server 又是 Client 仅 Host 才有的 UI
IsOwner 本客户端拥有该 NetworkObject 读输入、发移动 RPC
IsSpawned 已完成网络生成 避免 OnNetworkSpawn 前访问网络 API

IsServer && !IsClient → 专用服务器；IsServer && IsClient → Host。

核心组件

组件 作用
NetworkManager 连接、Spawn Player、Prefabs 列表、NetworkConfig
NetworkObject 可被复制到各端的实体；有 NetworkObjectId
NetworkBehaviour 挂在带 NetworkObject 的对象上；提供 RPC、NetworkVariable
NetworkTransform 同步 position / rotation / scale（含插值）
Unity Transport 底层 UDP（UTP），配置 IP、端口

层级要求：NetworkBehaviour 必须与 NetworkObject 在同一 GameObject，或在其子物体上（子物体上的脚本仍属该 NetworkObject）。

Pico 大空间 / VR

点 说明
地图原点 / 正方向 Pico 录制时标定的原点、朝向 → 映射到 Unity 场景原点与前进方向（常与 +Z 对齐，以项目文档为准）
VR Rig 头显、手柄父节点只驱动本地相机；不要把 Rig 根节点当网络 Player 直接同步
推荐结构 网络 Player（NetworkObject + NetworkTransform）← 子节点 Local XR Rig（仅 Owner 启用）
坐标系 大空间定位 SDK 给出的位姿需转换到 Unity 世界空间后再写入权威 Transform

public class NetworkedVRPlayer : NetworkBehaviour
{
    [SerializeField] Transform xrCamera; // 仅 Owner 本地引用

    void Update()
    {
        if (!IsOwner) return;
        // 采集本帧目标（例：摇杆 / 传送点）
        Vector3 target = ComputeTargetFromInput();
        SubmitMoveRpc(target);
    }

    [Rpc(SendTo.Server)]
    void SubmitMoveRpc(Vector3 worldPosition)
    {
        // 服务器校验（距离、冷却、是否在 NavMesh 上）
        if (!ValidateMove(worldPosition)) return;
        transform.position = worldPosition;
        // 若挂了 NetworkTransform，由组件自动同步；否则写 NetworkVariable
    }

    bool ValidateMove(Vector3 pos) => Vector3.Distance(transform.position, pos) < 5f;
}

    }
}

using Unity.Netcode;
using UnityEngine;

namespace HelloWorld
{
    public class HelloWorldPlayer : NetworkBehaviour
    {
        // 默认：Server 写，所有人读
        public NetworkVariable<Vector3> Position = new NetworkVariable<Vector3>(
            Vector3.zero,
            NetworkVariableReadPermission.Everyone,
            NetworkVariableWritePermission.Server);

        public override void OnNetworkSpawn()
        {
            Position.OnValueChanged += (_, newVal) => transform.position = newVal;
            if (IsOwner) Move();
        }

        public override void OnNetworkDespawn()
        {
            Position.OnValueChanged -= (_, newVal) => transform.position = newVal;
        }

        public void Move() => SubmitPositionRequestRpc();

        [Rpc(SendTo.Server)]
        private void SubmitPositionRequestRpc(RpcParams rpcParams = default)
        {
            var randomPosition = GetRandomPositionOnPlane();
            transform.position = randomPosition;
            Position.Value = randomPosition; // 仅 Server 可写
        }

        static Vector3 GetRandomPositionOnPlane()
        {
            return new Vector3(Random.Range(-3f, 3f), 1f, Random.Range(-3f, 3f));
        }
    }
}

生产环境更推荐用 OnValueChanged 更新 Transform，而不是每帧在 Update 里赋值（减少无谓逻辑）。

using Unity.Netcode;
using UnityEngine;

public class RpcTest : NetworkBehaviour
{
    public override void OnNetworkSpawn()
    {
        if (!IsServer && IsOwner)
            ServerOnlyRpc(0, NetworkObjectId);
    }

    [Rpc(SendTo.ClientsAndHost)]
    private void ClientAndHostRpc(int value, ulong sourceNetworkObjectId)
    {
        Debug.Log($"Client Received RPC #{value} on #{sourceNetworkObjectId}");
        if (IsOwner)
            ServerOnlyRpc(value + 1, sourceNetworkObjectId);
    }

    [Rpc(SendTo.Server)]
    private void ServerOnlyRpc(int value, ulong sourceNetworkObjectId)
    {
        Debug.Log($"Server Received RPC #{value} on #{sourceNetworkObjectId}");
        ClientAndHostRpc(value, sourceNetworkObjectId);
    }
}

同步方案选型

方式 优点 缺点 适用
NetworkTransform 自动插值、省代码 带宽随物体数量增 角色、可拾取物、大部分动态物体
NetworkVariable 精确、可订阅变化 需自己处理表现 血量、弹药、枚举状态、回合数
RPC 瞬时事件、无需持久状态 不保证顺序、不能当存档 开枪音效、一次性技能、聊天
手动 RPC + Transform 完全自定义 易出 desync、要写插值 特殊同步规则 
  flowchart TD
    A[需要同步什么?] --> B{持续 Transform?}
    B -->|是| C[NetworkTransform]
    B -->|否| D{离散状态?}
    D -->|是| E[NetworkVariable]
    D -->|否| F[一次性事件 → RPC]

    0f,
    NetworkVariableReadPermission.Everyone,
    NetworkVariableWritePermission.Owner);       // 仅 Owner 可写（仍经服务器转发）
写权限 说明
Server 只有服务器改 .Value（推荐：玩法真相在服）
Owner 拥有者客户端可发起写，实际仍由 NGO 同步规则处理

订阅变化（UI、动画触发）：

Health.OnValueChanged += (oldVal, newVal) =>
{

支持类型：基础值、Vector3、Quaternion、实现 INetworkSerializable 的自定义结构体等（见官方列表）。

注意

  • 不要在 OnNetworkSpawn 之前读写 .Value
  • 高频数据（每帧坐标）优先 NetworkTransform，不要滥用 NetworkVariable<Vector3>
  • 客户端不要用 NetworkVariable 做“仅本地”数据，用普通字段即可

RPC 详解

编写规则

规则 说明
声明位置 必须在 NetworkBehaviour 子类中
方法签名 通常 void，参数需为 NGO 支持的序列化类型
调用时机 仅在 IsSpawned == true 之后
谁可发起 SendTo.Server 一般由 Client/Owner 调；SendTo.Clients* 必须由 Server 调
安全 所有玩法判定在 Server RPC 内做，不要相信客户端传来的伤害、坐标

void CmdTakeDamageRpc(int damage, RpcParams p = default)

NetworkTransform 与移动

设置（概念） 说明
Server Authority 只有服务器改 Transform 会同步（NPC、物理球）
Owner Authority Owner 客户端驱动，适合第一人称 / VR 本地移动（仍需 Server 校验时用 RPC 改服）
插值 远端客户端平滑；本地 Owner 可关闭或减小插值避免“拖泥带水”

第一人称 / VR：常见组合 = Owner 读输入 + SendTo.Server 移动 RPC + NetworkTransform（Server 权威）或 Owner 权威 + 服务端校验。

    netObj.Despawn(); // 推荐：走网络销毁
// Destroy(go) 需知会网络层，否则易留幽灵对象
API 说明
Spawn() 创建网络实例，分配 NetworkObjectId
Despawn() 网络层销毁，各端 OnNetworkDespawn
ChangeOwnership(clientId) 转移 Owner（载具上下车、观战切换）
TrySetParent(parentNetObj) 网络父子关系（武器挂点等）

// NetworkManager 上挂脚本，或在自定义 Manager 里订阅
void Start()
{
    NetworkManager.Singleton.ConnectionApprovalCallback = ApprovalCheck;
    NetworkManager.Singleton.NetworkConfig.ConnectionApproval = true;
}

void ApprovalCheck(
    NetworkManager.ConnectionApprovalRequest request,
    NetworkManager.ConnectionApprovalResponse response)
{
    string payload = System.Text.Encoding.UTF8.GetString(request.Payload);
    bool approved = payload == "expected_token";
    response.Approved = approved;
    response.CreatePlayerObject = approved;
    response.Pending = false;
}

客户端连接时可通过 NetworkManager.Singleton.NetworkConfig.ConnectionData 传入自定义字节（版本号、房间密码等）。

场景与预制体

主题 说明
Network Prefabs List 所有会被 Spawn 的预制体必须注册，否则客户端缺资源
场景内 NetworkObject 场景中预先放的 NetworkObject 需在加载时由服务器同步生成（使用 NGO 场景管理或手动 Spawn）
DontDestroyOnLoad NetworkManager 常放启动场景并 DontDestroyOnLoad
Additive 场景 使用 NetworkSceneManager 做网络化场景加载（大空间分块可选）

调试与常见坑

现象 可能原因
RPC 不执行 未 Spawn、方法不是 NetworkBehaviour、Server RPC 在纯 Client 调
客户端看不到物体 预制体未进 Network Prefabs、Server 未 Spawn
位置抖动 / 回弹 Client 与 Server 同时改 Transform；应只权威端写
Host 正常、专服异常 用了仅 Client 的逻辑，未用 IsServer 区分
VR 身体与头分离 Rig 与 Network Player 根节点混用；应用子节点分离
变量不同步 写权限不是 Server 却在客户端改 .Value

调试工具

  • Multiplayer Play Mode：编辑器内多实例
  • Network Profiler / Stats：Package Manager → Multiplayer Tools
  • 日志：NetworkLog.LogInfo、连接事件 OnClientConnectedCallback

网络概念

主题 描述
权威（Authority） 谁负责该对象的“真相”状态；状态同步由权威端产生
所有权（Ownership） 哪个 ClientId 是该 NetworkObject 的 Owner；影响谁可发部分 RPC、Owner 写变量
网络拓扑 Dedicated Server、Listen Server（Host）、纯 P2P（NGO 以服为中心）

配置

连接配置

  • NetworkManager → Network Config：地址、端口、Player Prefab、Tick、连接数。
  • Connection Approval：ConnectionApproval = true 时必须实现 ConnectionApprovalCallback，否则连接会被拒。

传输（Transport）

  • Unity Transport（UTP）：UDP；可配 ConnectionData 的 Address、Port。
  • 局域网：127.0.0.1 + 同一端口；真机需本机局域网 IP。

Unity Relay

  • NGO + Relay：房间码 / Join Code 交换 Allocation 数据，客户端无需知道主机真实 IP。

命令行参数

  • -server、-client、-ip、-port 等由启动脚本解析并写入 UnityTransport。
